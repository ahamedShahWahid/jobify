# P3.1 — Notifications outbox + email channel (stubbed) — design

**Date:** 2026-05-20
**Spec ref:** `IMPLEMENTATION_SPEC.md` §5 (`notifications` table), §6.4 (outbox pattern + channels), §13 P3 ("notifications + applications").
**Status:** Draft — pending approval.

Second and final P3 slice. P3.0 lets the applicant apply, save, withdraw. This slice ships the notification rail: an apply triggers a "Application received" notification; an outbox sweeper dispatches pending rows; an in-app inbox endpoint surfaces them. Email channel is **stubbed** (logs only) — the adapter is a Protocol so real SES drops in cleanly when the deploy target is picked.

## Why this slice

- **The BRD's user journey ends with "candidates know what's happening".** Without notifications, an applicant has no feedback after clicking Apply.
- **The outbox pattern is the spec's prescribed shape.** Insert-then-sweep separates writers from channel adapters, which makes async retries and channel failures local concerns.
- **Stubbed email today, real SES later.** A Protocol-based channel adapter lets us defer the SES integration without changing the writer or sweeper code paths.

## Decisions resolved

| # | Decision | Resolution |
|---|---|---|
| 1 | Channels in P3.1 | **Email (stubbed) + in-app inbox.** Web push deferred (web-only MVP). WhatsApp deferred per spec §6.4 + BRD #5. |
| 2 | Email adapter | **Protocol-based** (`EmailChannel`) with two impls: `LoggingEmailChannel` (stub for MVP — logs the would-be payload via structlog) and a future `SESEmailChannel` (boto3, deferred). Selected via `KPA_EMAIL_CHANNEL` env var: `logging` (default) or `ses`. |
| 3 | Outbox sweeper | **Celery task `sweep_notifications`** on a new `notify` queue. Beat scheduling deferred — invoked from tests directly and runnable on demand. Beat-integration is a follow-up when the worker infra hardens. |
| 4 | Sweep semantics | **`SELECT ... FOR UPDATE SKIP LOCKED LIMIT N`** over `status='pending' AND send_after <= now() AND deleted_at IS NULL`. Default batch size 50, configurable via `KPA_NOTIFY_BATCH_SIZE`. |
| 5 | Per-row state machine | `pending → dispatching → sent | failed`. Sweeper sets `dispatching` before the channel call; on success → `sent` + `sent_at`; on failure → `failed` + `last_error` + `attempts++` + `send_after = now() + backoff`. Max 5 attempts; final failure leaves the row at `failed` for admin inspection. |
| 6 | Retry backoff | Exponential with jitter: `min(60s * 2^(attempts-1), 1h) + random(0..30s)`. Pure-Python computation; no Celery autoretry on the sweeper itself (it's a fan-out). |
| 7 | Channel adapter contract | `async def send(notification: Notification) -> ChannelResult` — returns `ChannelResult.ok()` or `ChannelResult.failed(message)`. Adapter never raises on expected failures (network blip, 4xx); raises only on programming errors. |
| 8 | Trigger from apply endpoint | **Insert a `pending` notification on every successful 201 (new application).** Idempotent re-applies (200) do NOT trigger a new notification. Re-applies after withdraw also don't (treated as the same logical application). |
| 9 | Notification payload | JSONB `payload` column. Shape per kind. For `application_received`: `{kind: "application_received", application_id, job_id, job_title, employer_name, applicant_email}`. Future kinds extend the union. |
| 10 | In-app inbox | `GET /v1/notifications` returns `pending`, `dispatching`, `sent` rows for the current user. `failed` rows are admin-only (hidden from the inbox). Cursor pagination + ETag, same shape as feed/applications. |
| 11 | Mark-read | `POST /v1/notifications/{id}/read` sets `read_at = now()`. Idempotent. Only applicant's own rows. |
| 12 | Status enum scope | `pending`, `dispatching`, `sent`, `failed`. No `cancelled` — soft-delete via `deleted_at` if needed. |
| 13 | Per-user channel preferences | **Out of scope.** Spec §6.4 mentions `consents` table; that lands with P4 (DPDP). For P3.1, email is always sent for the one notification kind we have. |

## Schema

### `notifications` (migration 0011)

```python
class NotificationStatus(StrEnum):
    PENDING = "pending"
    DISPATCHING = "dispatching"
    SENT = "sent"
    FAILED = "failed"


class NotificationChannel(StrEnum):
    EMAIL = "email"
    IN_APP = "in_app"


class Notification(Base):
    __tablename__ = "notifications"

    id: Mapped[UuidPK]
    user_id: Mapped[uuid.UUID] = mapped_column(
        UUID(as_uuid=True),
        ForeignKey("kpa.users.id", ondelete="CASCADE"),
        nullable=False,
    )
    kind: Mapped[str] = mapped_column(String(64), nullable=False)
    channel: Mapped[NotificationChannel] = mapped_column(
        SAEnum(NotificationChannel, name="notification_channel", schema="kpa",
               native_enum=True, values_callable=lambda x: [e.value for e in x]),
        nullable=False,
    )
    status: Mapped[NotificationStatus] = mapped_column(
        SAEnum(NotificationStatus, name="notification_status", schema="kpa",
               native_enum=True, values_callable=lambda x: [e.value for e in x]),
        nullable=False,
        default=NotificationStatus.PENDING,
        server_default=NotificationStatus.PENDING.value,
    )
    payload: Mapped[dict[str, Any]] = mapped_column(JSONB, nullable=False)
    send_after: Mapped[datetime] = mapped_column(
        DateTime(timezone=True), server_default=func.now(), nullable=False
    )
    sent_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)
    read_at: Mapped[datetime | None] = mapped_column(DateTime(timezone=True), nullable=True)
    attempts: Mapped[int] = mapped_column(Integer, nullable=False, default=0, server_default="0")
    last_error: Mapped[str | None] = mapped_column(Text, nullable=True)
    created_at: Mapped[CreatedAt]
    updated_at: Mapped[UpdatedAt]
    deleted_at: Mapped[DeletedAt]

    __table_args__ = (
        # Sweeper query path
        Index(
            "ix_notifications_status_send_after_live",
            "status", "send_after",
            postgresql_where="deleted_at IS NULL AND status IN ('pending', 'dispatching')",
        ),
        # User inbox query path
        Index(
            "ix_notifications_user_id_created_at_live",
            "user_id",
            text("created_at DESC"),
            postgresql_where="deleted_at IS NULL",
        ),
        {"schema": "kpa"},
    )
```

## Channel adapter

`api/src/kpa/integrations/notifications/base.py`:

```python
from typing import Protocol


class ChannelError(Exception):
    """Recoverable error from a channel adapter. Sweeper will retry."""


@dataclass(frozen=True)
class ChannelResult:
    ok: bool
    message: str = ""

    @classmethod
    def success(cls) -> "ChannelResult":
        return cls(ok=True)

    @classmethod
    def failed(cls, message: str) -> "ChannelResult":
        return cls(ok=False, message=message)


class EmailChannel(Protocol):
    async def send(self, notification: Notification, *, recipient: str) -> ChannelResult: ...
```

## Sweeper task

`api/src/kpa/workers/tasks/sweep_notifications.py`:

```python
@celery_app.task(name="kpa.sweep_notifications", bind=True, acks_late=True)
def sweep_notifications(self) -> None:
    _run(lambda: _sweep_notifications_async())


async def _sweep_notifications_async() -> None:
    sm = get_session_maker()
    settings = Settings()
    email_channel = get_email_channel()

    async with sm() as session:
        # SELECT FOR UPDATE SKIP LOCKED — claim a batch
        stmt = (
            select(Notification)
            .where(
                Notification.deleted_at.is_(None),
                Notification.status == NotificationStatus.PENDING,
                Notification.send_after <= func.now(),
            )
            .order_by(Notification.send_after.asc())
            .limit(settings.notify_batch_size)
            .with_for_update(skip_locked=True)
        )
        rows = (await session.execute(stmt)).scalars().all()
        # Transition to dispatching atomically
        for n in rows:
            n.status = NotificationStatus.DISPATCHING
        await session.commit()

    for notification in rows:
        try:
            result = await _dispatch_one(session_maker=sm, email_channel=email_channel,
                                          notification_id=notification.id)
            # _dispatch_one handles state transition + commit
        except Exception:
            _log.exception("sweep.dispatch-unexpected", notification_id=str(notification.id))


async def _dispatch_one(*, session_maker, email_channel, notification_id) -> None:
    async with session_maker() as session:
        n = await session.get(Notification, notification_id)
        if n is None or n.deleted_at is not None:
            return
        # Re-read recipient: for now we look up users.email; in future, payload may carry it.
        user = await session.get(User, n.user_id)
        if user is None or user.email is None:
            n.status = NotificationStatus.FAILED
            n.last_error = "user_missing_or_no_email"
            await session.commit()
            return

        if n.channel == NotificationChannel.EMAIL:
            result = await email_channel.send(n, recipient=user.email)
        elif n.channel == NotificationChannel.IN_APP:
            # In-app delivery is "the row exists" — mark sent immediately.
            result = ChannelResult.success()
        else:
            result = ChannelResult.failed(f"unknown_channel: {n.channel}")

        if result.ok:
            n.status = NotificationStatus.SENT
            n.sent_at = func.now()
            n.last_error = None
        else:
            n.attempts += 1
            n.last_error = result.message
            if n.attempts >= 5:
                n.status = NotificationStatus.FAILED
            else:
                n.status = NotificationStatus.PENDING
                delay = min(60 * (2 ** (n.attempts - 1)), 3600) + random.randint(0, 30)
                n.send_after = datetime.now(UTC) + timedelta(seconds=delay)
        await session.commit()
```

## Endpoints

### `GET /v1/notifications`

- Auth: `current_user` (any role — notifications can target recruiters/admins later too). For P3.1, `_require_applicant` matches the rest of the surface; expand when other roles get notifications.
- Returns: `{items: NotificationRead[], next_cursor}` — list of `pending`, `dispatching`, `sent` rows for current user. `failed` excluded.
- Cursor: `{created_at, notification_id}` base64.
- ETag.

### `POST /v1/notifications/{id}/read`

- Auth.
- Load by id scoped to current user + `deleted_at IS NULL`. Missing → 404 `notification_not_found`.
- Set `read_at = now()` if currently NULL. Idempotent (re-call is no-op).
- Response: 200 `NotificationRead`.

## Out of scope (intentional)

- **SES integration** — deferred until deploy target picked. `LoggingEmailChannel` is the only impl.
- **Celery beat scheduling** — sweeper is callable on demand. Beat config lands with worker hardening.
- **FCM / WhatsApp channels** — deferred per the agreed P2/P3 + web-only scope.
- **Per-user channel preferences (consents)** — P4 (DPDP).
- **Notification templates / i18n** — payload is structured; rendering to email body is the future SES channel's concern.
- **Admin-side view of `failed` notifications** — covered by future admin slice.

## Risks

- **`FOR UPDATE SKIP LOCKED` requires Postgres ≥ 9.5** — local Postgres 16 is fine.
- **Sweeper running concurrently with itself** — `SKIP LOCKED` makes this safe (two workers claim disjoint batches). Beat single-replica is still the production deployment target.
- **`LoggingEmailChannel` does nothing useful** — emails aren't really sent. Acceptable for dev/MVP; documented prominently in README + CLAUDE.md.
- **Notification insertion on apply ties two concerns to one commit** — if the notification insert fails, the application also rolls back. Reasonable for MVP (rare; notifications table is local). At production scale, consider INSERT-after-commit with a fire-and-forget pattern (like the embed dispatch wiring).
