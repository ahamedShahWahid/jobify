# Jobify Architecture Review â€” July 2026

**Lens:** near-term shipping velocity (solo dev + Claude). **Scope:** whole repo â€” `core/`, `api/`, `worker/`, `tests/`, `app/` (Flutter), `frontend/` (React/Vite).
**Method:** 5 rubric-driven explorer agents (one per surface) + main-session cross-cutting synthesis. Every finding below was re-read first-hand from the cited file before being included; a few explorer-reported nits with clean, specific citations were kept without a second independent read (noted inline where relevant).

## 1. Verdict

The architecture is in good shape for its stage: dependency direction is genuinely clean (verified â€” `core` has zero imports from `api`/`worker`, and `api`/`worker` never import each other), the domain-invariant-heavy stuff (soft-delete, DSR coverage, migration hygiene) is enforced by tests rather than convention, and the three client-facing packages (`api`, `app`, `frontend`) each have a real HTTP/business-logic or shared/per-surface split that is actually honored in the code, not just claimed. The risk isn't structural rot â€” it's that the **documentation layer has started to outrun the update discipline that made it trustworthy**: every single surface's explorer found at least one CLAUDE.md/README/spec claim that no longer matches the code, and one surface (`frontend/`) has a live bug (dead anchor links) sitting inside code whose own CLAUDE.md documents the exact anti-pattern as the rule to follow. The other dominant theme is the **manually-mirrored contract** between the backend's OpenAPI surface and two hand-written clients (Dart, TypeScript) â€” currently held together by spot-checks and one good fixture-pin pattern (`MeDto`), not by anything systematic. Neither theme requires urgent architectural surgery; both are cheap to address now and expensive to leave for the next five slices to trip over.

## 2. What's working â€” keep these

- **Clean, verified dependency graph.** `core/src/jobify/**` has zero imports from `api` or `worker` (grep-confirmed by the core explorer); `api/` has zero dependency on `jobify-worker` (confirmed no `jobify_worker` import in `api/src`, and `api/pyproject.toml` doesn't list it) â€” `enqueue()` (`core/src/jobify/celery_app.py:32-46`) is the only coupling point. This is a real boundary, not aspirational: it keeps `api`'s build/test surface free of worker-only dependencies (e.g. `google-genai`).
- **Soft-delete is enforced by structure, not convention.** `tests/unit/test_soft_delete_invariant.py` (`test_every_domain_model_has_soft_delete_column`, `test_exemptions_are_real_mapped_classes`) mechanically asserts every domain model carries `deleted_at`. This is exactly the kind of guard that survives a developer forgetting, rather than relying on code review catching it every time.
- **The PII-table growth path is closed by a contract-pin test.** `tests/unit/dsr/test_dsr_coverage.py` (`test_export_covers_exactly_the_expected_pii_tables`, `test_deleter_covers_exactly_the_expected_pii_tables`) fails loudly if a new PII-bearing table isn't wired into both DSR export and delete â€” the single most common way this class of app silently leaks data on a new table.
- **HTTP-layer / business-logic split in `api/` is real, not just documented.** `routes/dsr.py` (155 lines, pure HTTP glue) delegates to `jobify_api/dsr/__init__.py` + `dsr/deleter.py`; `routes/employers/*` delegates role logic to `jobify_api/employers/membership.py`. One home for a reusable rule beats re-deriving it per route.
- **The 3-transaction split in worker tasks is applied consistently and correctly.** `parse.py`, `embed.py`, `embed_job.py` all gate â†’ make the external call â†’ verify-and-write in separate transactions, so external I/O (storage reads, Gemini calls) never happens under a held row lock â€” verified in all three files.
- **Migration hygiene is unusually good for a one-person MVP.** All 20 files under `core/src/jobify/db/migrations/versions/` define both `upgrade` and `downgrade`; autogenerate is deliberately off; non-obvious index choices are documented inline (e.g. the HNSW/cosine choice in `0004`, the raw-SQL DESC index in `0007`).
- **Flutter's `core/` is a genuinely clean leaf** â€” zero imports from `data/`/`presentation/` (grep-confirmed) â€” and the `MeDto` DTO-drift mitigation (inline "mirrors backend file X, verified against test Y" comments + a literal-JSON fixture test) is a cheap, real substitute for codegen. Worth extending to more DTOs, not replacing.
- **Frontend transport/session layering is well-factored and actually shared.** `src/shared/api/transport.ts` (`BaseHttpClient`, single-flight 401 refresh) and `src/shared/session/createSession.tsx` are instantiated per surface with zero copy-pasted auth logic â€” verified no `shared/*` file imports from `sites/*` anywhere in the tree.
- **India-first formatting has one real source of truth.** `frontend/src/shared/format.ts` is delegated to correctly even where it would have been easy to hand-copy (`console/components/bits.tsx` re-exports `inrLakh` rather than reimplementing it).

## 4. Roadmap

Ordered by payoff-per-effort under the velocity lens, not raw severity â€” the point is what to do *next*, not a threat ranking.

1. **Fix the dead anchor links** (finding #1). **Size: S.** Swap the 7 cited `<a href="/#...">` tags for the `<Link to="/<surface>#...">` form already used one branch over in the same files. **Payoff:** stops shipping a live wrong-surface-navigation bug; the fix pattern already exists in the same file, so this is closer to "finish a migration" than "design a fix." **Skip-cost:** every employer/applicant who clicks a footer link keeps landing on the wrong surface.

2. **De-duplicate the role-flip logic** (finding #7). **Size: S.** Replace `routes/employers/core.py:86-90`'s inline `update()` with a call to `flip_to_recruiter()`. **Payoff:** one fewer place a future membership-logic change has to remember to touch â€” the exact failure shape `api/CLAUDE.md` already tells a war story about (`require_applicant`, 7 copies, one silently lost a filter). **Skip-cost:** guaranteed eventual drift between the two copies, discovered the hard way.

3. **Batch-fix the confirmed doc-drift items** (finding #4). **Size: S** (a documentation-only pass, ~6 files). **Payoff:** restores the CLAUDE.md set's trustworthiness â€” for a solo dev + Claude, "read the package CLAUDE.md and trust it" is the main leverage this repo's doc investment buys; every unfixed drift instance is a future session's wasted turn (or worse, a wrong assumption baked into new code). **Skip-cost:** compounds â€” new sessions keep trusting a document that's already been shown wrong in 5/5 surfaces reviewed.

4. **Decide the fate of frontend's `ds-*` shared CSS layer** (finding #5). **Size: S to delete, M to actually migrate one surface onto it.** Recommend deleting `components.css`'s unused classes now (verified zero consumers) rather than migrating â€” migration is a real M-effort investment that should wait until a second surface change actually needs shared components, proven by a real slice, not speculatively. **Payoff:** stops maintaining a fourth (dead) button implementation alongside the three real (triplicated) ones. **Skip-cost:** the next contributor who finds `components.css` and trusts CLAUDE.md's "canonical baseline" claim will style a new feature against dead code.

5. **Correct `IMPLEMENTATION_SPEC.md`'s app-client section** (finding #3). **Size: S**, doc edit only â€” either delete the codegen/retry/telemetry claims or explicitly mark them "not yet built, backlog." **Payoff:** the next planning session won't assume a safety net exists that doesn't, and won't waste a slice re-discovering interceptors that were never actually shipped. **Skip-cost:** low-frequency but expensive when it hits â€” exactly the kind of thing that costs a full debugging session before someone realizes the doc was aspirational.

6. **Extend the `MeDto` fixture-pin pattern to `JobStatus`/`ApplicationStatus`** (finding #9). **Size: S.** Add the same "mirrors backend file X" comment + literal-fixture round-trip test already proven for `MeDto`. **Payoff:** closes the cheapest, highest-value slice of the larger contract-sync gap (finding #2) without building anything new â€” the pattern already exists and works. **Skip-cost:** a backend enum value addition silently degrades to `unknownEnumValue` in the app instead of failing a test.

7. **Decide sweep_notifications' fate** (finding #10). **Size: M** â€” either wire a real trigger (Celery beat entry, or an admin-triggerable endpoint) or explicitly descope/flag the notification-outbox feature as "written but not yet sending" until prioritized. **Payoff:** closes a "looks done, isn't" gap before a slice builds more features on top of an assumption that outbox rows get delivered. **Skip-cost:** growing amount of dead functionality (routes writing to an outbox nothing reads) the longer it's left implicit.

8. **Consolidate `score_applicant.py`/`score_job.py`'s shared shape** (finding #6). **Size: M.** Extract the shared batch-pagination and UPSERT-with-`coalesce` logic into one function both tasks call. **Payoff:** halves the surface area of the single highest-semantic-risk duplication in the review â€” a scoring-logic fix currently has to be applied twice with nothing forcing parity. **Skip-cost:** the two files will eventually diverge on a real scoring-semantics change, and existing tests (per-task, not shared) won't catch the divergence.

9. *(Larger, optional â€” not urgent)* **A lightweight contract-sync check.** **Size: L.** A script diffing `openapi_snapshot.json`'s schema field names against the fixture-pinned Dart/TS DTOs, failing CI when a backend field changes with no matching client update. **Payoff:** closes the actual root cause behind findings #2 and #9 at the systemic level rather than one DTO at a time (finding #3 is a separate doc-accuracy problem, already covered by roadmap #5). **Skip-cost of deferring:** low today (40 paths / 62 schemas, one incident-free history so far) â€” this is the "big rock" to revisit if a real contract-break bug ships, not before.

## 5. Non-recommendations

- **Don't build the full OpenAPI-to-Dart/TS codegen pipeline `IMPLEMENTATION_SPEC.md` describes, at least not now.** At 40 paths / 62 schemas and a one-developer team, the `MeDto`-style fixture-pin pattern is already catching drift at near-zero build/toolchain cost. Full codegen adds a generator, generated-file review discipline, and version-pinning across three languages â€” real ongoing cost â€” for a failure rate this review found to be one stale comment and two under-covered enums. **Signal that would flip this:** a real production incident traced to an undetected contract break, or the schema count roughly doubling.
- **Don't split `core/src/jobify/db/models.py` into per-entity files.** Explicitly reviewed and found not to be a problem: 954 lines but ~16 independent classes; each edit touches ~30 contiguous lines; FKs are already string-typed (`ForeignKey("jobify.users.id")`), so a future split stays cheap to do later. Splitting now is effort spent on a risk that hasn't materialized.
- **Don't resurrect the Flutter `domain/` layer to match the stale 2026-05-21 spec.** The current `data/presentation/core` split with interfaces living next to their impls is working â€” proven by `test/helpers/fake_*_repositories.dart` injecting fakes directly against those interfaces in real tests. The `domain/` layer was a design that was deliberately simplified away, not an oversight; fix the stale spec pointer (roadmap #3), don't rebuild the layer it described.
- **Don't consolidate `api/`'s single-file packages** (e.g. the 0-byte `employers/__init__.py` wrapping just `membership.py`) **into flat modules.** Cosmetic nit only; a repo-wide pass here is mechanical churn across many files for zero velocity payoff.
- **Don't pursue further service extraction between `api`/`worker`/`core`.** The current boundary (core owns the domain, api and worker both depend on it, api and worker never depend on each other) is real and working. Nothing in team size, deploy cadence, or observed scaling pain justifies more separation right now.
- **Don't build a full frontend test suite or ESLint config as a blanket investment.** A real gap (zero tests, no lint config), but at solo-dev velocity a wholesale suite competes directly with feature-shipping time. The one piece of logic worth a *targeted* regression test is the 401 single-flight refresh ladder in `src/shared/api/transport.ts:96-140` â€” the riskiest untested code found in the review â€” not the whole surface.
