# CLAUDE.md — Flutter app (`app/`)

Load-bearing invariants for the Flutter client. Auto-loaded when working under `app/`; the root `CLAUDE.md` holds the backend invariants and repo overview.

Applicant iOS + Android + Web. `lib/data/` + `lib/presentation/` + `lib/core/` (no `domain/`); abstract repo interfaces next to impls (`data/<feature>/<repo>_repository.dart` + `_impl.dart`). Riverpod 4.x (code-gen), dio 5.7, go_router 14.6 with `StatefulShellRoute.indexedStack` (four-tab nav). Run/test commands + web-OAuth origin setup: `app/README.md`. Re-run `dart run build_runner build --delete-conflicting-outputs` after touching `@freezed`/`@riverpod`/`@JsonSerializable`.

**Design standard: `docs/design-system.md`** — the full catalogue (Clear Sky palette, typography ramp, radii, brand canvas, shared philosophy, hard rules). Read it before styling any screen.

## Non-obvious bits

- **Refresh-on-401 interceptor** (`lib/data/api/refresh_on_401_interceptor.dart`) is the most important code: single-flight via `Completer<String>?`. Tests are the canonical spec — keep passing. **Cleanup order:** `_inFlight = null` BEFORE `complete()` (else a synchronous continuation re-enters `onError` on a stale completer).
- **Don't override `validateStatus` on Dio** — `(s) => s < 500` masks 401s so the refresh interceptor never fires (tests pass because the mock maps 401→reject). Default Dio (4xx/5xx throws) is required.
- **Auth composition root** (`lib/app/composition_root.dart`) constructs Dio, `AccessTokenHolder`, and `AuthRepository` as one object graph so data-layer providers never import each other or presentation. The legacy provider paths are compatibility exports only. Auth-state callbacks are wired at this top-level boundary.
- **Riverpod 4.x codegen drops the `Notifier` suffix** — `AuthStateNotifier` → `authStateProvider`.
- **No feed mutation on apply/save/withdraw/unsave** — each invalidates the relevant list controller + `jobDetailControllerProvider(id)`, never the feed (`feedControllerProvider`, the job list). They now ALSO invalidate `feedSummaryControllerProvider` — the Feed home-summary's Applications/Saved counts are a *separate* aggregate provider, and since `FeedScreen` lives in the applicant shell's `StatefulShellRoute.indexedStack`, it (and anything it watches) is kept alive for the whole session by go_router's `AutomaticKeepAliveClientMixin` — never disposed on tab switch, so a bare `autoDispose` provider never "naturally" refreshes on tab re-entry. Any future mutation that changes applications/saved counts must invalidate `feedSummaryControllerProvider` too, or its tiles go stale until the app restarts. `FeedScreen`'s header refresh + pull-to-refresh call a combined `_refreshAll()` that refreshes both `feedControllerProvider` and `feedSummaryControllerProvider`. Match-feedback mutations are the documented exception to "no feed mutation on apply/save/withdraw/unsave": `FeedController.rateDown/rateUp/undoDown` mutate the feed's list state in place (rateDown/rateUp patch `state` directly; `undoDown` clears server-side then refetches), and `MatchFeedbackController` (job detail) invalidates `feedControllerProvider` because a down-rate changes feed membership server-side.
- **List screens share `PagedState<T>` + `loadNextPage`** (`lib/presentation/paging/`). The error path preserves loaded items via `AsyncValue.error(...).copyWithPrevious(...)` — `copyWithPrevious` is `@internal`; the `// ignore: invalid_use_of_internal_member` in `paging.dart` is load-bearing.
- **`preferencesControllerProvider` is `keepAlive` and SHARED** (ProfileScreen, FeedSummaryRow's match-profile tile, EditProfileScreen, the ResumeScreen→PreferencesScreen flow). Three rules that each fixed a real bug: (a) **every path that pushes `SignedOut` must `ref.invalidate` it** — sign-out AND delete-account — or the next session leaks the previous user's role/locations/CTC; (b) `submit()` preserves loaded data via `copyWithPrevious` — a bare `AsyncError` on a failed save radiates to every consumer; (c) consumers branch on the full `AsyncValue` — `.value == null` conflates loading/error with "no data" (the nudge-banner false-flash + the edit-screen seeding-empty-then-wiping-locations bugs). Forms seed lazily once `hasValue` and are gated until then.
- **`PreferencesUpdateDto` is FULL-FORM** (always sends all three keys; explicit null = the backend's "clear"; empty `locations` list clears). `DesiredRole.unknown` (unrecognised server value) is **omitted from toJson** so a save preserves the server's value — never emit `desired_role: null` from `unknown` (`unknown.wireValue` throws by design); dropdowns coerce `unknown` → null for display but keep the raw value in state. `toJson` is hand-written (json_serializable emits Dart-3.8-only null-aware elements under this SDK pin — see the DTO's doc comment); the round-trip unit tests in `test/unit/data/preferences/` pin the hand-maintained wire maps.
- **Magic strings in enums** (`JobStatus`/`ApplicationStatus`/`ApplicationSource`/`MatchGenerator` use `@JsonKey(unknownEnumValue: X.unknown)`; slugs in `lib/core/error/auth_slugs.dart`). DTOs are `@JsonSerializable` plain by default; `@freezed` only for `copyWith` (`JobDetailDto` + `PagedState<T>`).
- **`/v1/me` is nullable on `email` + `applicant.full_name`** (DB columns are nullable — DSR scrubbing / phone-only auth later); `MeDto` mirrors this with `String?`. Keep DTO nullability in lockstep with the Pydantic response models, not with the happy path.
- **Per-tab nav stacks** — `/jobs/:id` is a child under each branch; JobDetail's 404 uses `context.pop()`.
- **Widget tests use `ThemeData.light(useMaterial3: true)`, NOT `buildTheme()`** (`buildTheme` calls `GoogleFonts.inter(...)` → network fetch, fails offline/CI).
- **`PackageInfo.fromPlatform()` in a `keepAlive: true` provider**, not `FutureBuilder.future:` (which re-runs the platform-channel per rebuild). Shared module-static date formatters live in `core/format/date_formats.dart` (ICU parse at construction).
- **`--dart-define`, no flavors:** `JOBIFY_API_BASE_URL` + `JOBIFY_GOOGLE_WEB_CLIENT_ID` required at compile time (`Env.validateOrThrow()` in `main()`). Light theme only in v0. Shared test infra in `test/helpers/` (`MockInterceptor`, `fake_repositories.dart`).
- **Google sign-in is two flows** (detail: `specs/2026-05-21-flutter-app-shell-design.md`). Mobile: imperative `GoogleSignInDataSource.getIdToken()`. Web: GIS `signIn()` returns no `idToken`, so `google_web_sign_in.dart` uses `renderButton()` via `googleWebSignInProvider` (`keepAlive FutureProvider` awaiting `initialize()`); impl by conditional import (`_web.dart` / `_stub.dart` keeps `dart:js_interop` off mobile/test); `SignInScreen` branches on `kIsWeb`. `completeWebSignIn(idToken)` lives on the impl, reached via downcast. `GoogleSignInDataSourceImpl` builds `GoogleSignIn` platform-conditionally (`clientId:` web / `serverClientId:` mobile).
- **Local web sign-in** needs an Authorized-JS-origin on the web OAuth client + API CORS (`JOBIFY_CORS_ALLOW_ORIGINS`) — full setup + curl probe in `app/README.md` ("Web Google sign-in").
- **Privacy screen** (`presentation/privacy/`) = consent toggles + DSR export + DSR delete nav. Reserved scopes (`whatsapp_notifications` etc.) deliberately HIDDEN (no Dart gate yet). DSR export uses the clipboard in v0. DSR delete is a separate screen `/profile/privacy/delete` with a `DELETE_MY_ACCOUNT` guard; success → clear `AccessTokenHolder`, push `SignedOut`. Turning OFF `email_transactional` triggers a confirmation dialog.
- **Recruiter feature** (`presentation/recruiter/` + `data/jobs/recruiter_*` + `data/employers/team/`; spec `2026-06-06-recruiter-employer-experience-design.md`). Second `StatefulShellRoute` under `/recruiter/*`; `roleAwareRedirect` (`role_redirect.dart`) flips shells on `SignedIn.role`. Team tab (`recruiter_employer_screen.dart`) derives owner-ness from the caller's own roster row via `SignedIn.userId` (UI gating is defense-in-depth) and HIDES change-role/remove on your own row. Invitee `PendingInvitesScreen` (`/profile/invites`) accepts → `refreshSession()` → role flips → redirect to recruiter shell; **`accept()` treats `acceptInvite` as the success boundary** (a failed post-accept refresh is swallowed — re-accepting 404s). `/recruiter/jobs/:id/edit` with null `extra` resolves the job from the include-closed list via `EditJobResolver` (no blank create form). Dashboard sums `listMyJobs(status:'closed')` (open+closed) client-side.
- **Applicant profile redesign + Feed home summary** (PR #60, spec+plan at `docs/superpowers/specs/2026-07-06-applicant-feed-home-design.md` / `docs/superpowers/plans/2026-07-06-applicant-feed-home.md`). Profile/Edit/Preferences screens use a "spec sheet" visual language — label left (Inter, quiet), value right (`JobifyTypography.mono`, right-aligned) — with caveat-amber (`JobifyColors.caveatLight/Dark`) tappable "Add" prompts in place of a dead dash for the three fields `PreferencesDto.isComplete` tracks (desired role, locations, expected CTC). `FeedSummaryController`/`FeedSummaryRow`/`FeedVisitRepository` are the Feed-side pieces (see the invalidation-on-mutation bullet above); `FeedSummaryController.build()` uses `Future.wait` (not sequential awaits) fetching Applications+Saved — sequential awaits would leave the second future's rejection unobserved if the first throws first.
- **Riverpod 3.1.0 providers auto-retry on error by default** — `ProviderContainer.defaultRetry` (10 retries, 200ms→6.4s exponential backoff, ~34s worst case), skipped only for `ProviderException`/`Error` (a plain `throw Exception(...)` in a fake repo does NOT qualify). A test that makes a provider's `build()` throw and awaits `provider.future` will otherwise hang ~30s before failing. Fix: construct the test's `ProviderContainer(retry: (_, __) => null)` to disable retry for that container. Applies to any provider in this app, not just Feed's.
