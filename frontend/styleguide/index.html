Overview

One platform · three audiences · three voices

Jobify ships three distinct frontends. Each speaks to a different audience in a deliberately different voice — yet all are built on the same structural ideas: hand-authored CSS-variable token systems, no UI framework, and a single orchestrated load reveal. This catalogue is the shared reference: the tokens themselves live in each app’s own :root block.

P1

Tokens, not theming

Every surface, ink, accent and hairline is a CSS custom property in a single :root block per app. The token is the source of truth — this page only mirrors it.

P2

No framework

Plain, additive CSS. Components are class-composed, not imported; the cascade is the design system. Nothing to upgrade, nothing to fight.

P3

One reveal

A single staggered rise animation on load, guarded by prefers-reduced-motion. Motion sets the rhythm; it is never load-bearing.

P4

Distinct voice

A control room, a broadsheet, a prospectus. Each property earns its own typefaces and palette so the audience always knows which room they are in.

B0 · Brand

One brand, three voices

Each property paints its own room — but they all answer to a single mark and a single colour. Jobify Blue is the one constant a candidate, a recruiter and an admin recognise, whichever surface they are on. It lives as a --brand-blue token in every app’s :root, above each property’s own accent.

Colophon

Each property owns a single CSS file with a :root token block at the top — surfaces, ink layers, accents, hairlines, and semantic ok / danger states. Everything below is additive, hand-authored, framework-free CSS; sections are appended over time rather than refactored.

This catalogue is a hand-maintained snapshot. The tokens are authoritative in the apps — when a value changes in console.css, web/site.css, or employers/site.css, this page is updated to match. It is documentation, not a build output.

  • web/src/styles/site.css

Conventions

  • :root custom properties
  • No UI framework
  • Additive section CSS
