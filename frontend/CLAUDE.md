# CLAUDE.md — frontend (unified Vite + React + TS web app)

Load-bearing invariants for the web app: two route-prefixed surfaces under one HashRouter — `/employers` (recruiter marketing + authenticated recruiter workspace, `src/sites/employers`), `/console` (jobify-internal admin ops only, `src/sites/console`); shared transport/session/auth/env in `src/shared`. `/` redirects to `/employers`. Auto-loaded when working under `frontend/`. Dev/build/env reference is in `frontend/README.md`.

**The applicant web surface (`src/sites/web`) was removed 2026-07** — the Flutter app (`app/`) is the applicant client. If a future task needs an applicant browser surface again, it's recoverable from git history, not to be rebuilt from scratch speculatively.

**Recruiter ops moved from `/console` to `/employers` (2026-07)** — console is jobify-internal (admin) only now; a recruiter should never reach it. Recruiter pages live at `src/sites/employers/pages/dashboard/`, with their own `session.tsx` and `api/client.ts` independent from console's. Console's routes/links are built from `CONSOLE_BASE` (`src/sites/console/base.ts`) rather than hardcoded `/console/...` strings, so it's ready to move to its own subdomain later.

**No LLM/AI API calls from the client — ever.** All AI work (parsing, embeddings, match explanations) is backend-only, behind the Jobify REST API: the `google-genai` SDK lives in `core` and runs in the worker. The client never holds an AI-provider key and never contacts an AI host; the only external host it may contact is `accounts.google.com` (sign-in). Client-side "LLM" strings are metadata/copy only (the `generator` provenance tag, marketing stamps). A slice that needs new AI behavior adds a backend endpoint, not a client-side call.

> These are the shared-HashRouter pitfalls — things that silently send users to the wrong surface or leak styles across surfaces.

- **One app, two sibling route subtrees** — `index.html` → `src/main.tsx` → `src/App.tsx` mounts each surface's route fragment under one `<HashRouter>`. Both ship in one static `dist/`; they are NOT separate builds. Each surface mounts its own `SessionProvider` (sessions are independent per surface).
- **Every internal nav literal must carry the surface prefix.** Grep for ALL forms when migrating/adding routes — `grep -rnE "(to=\{?[\"'\`]|navigate\([\"'\`])/(seg)" src` — backtick-template ``to={`/console/${id}`}`` and single-quote `navigate('/console')` slip past quote-only greps.
- **Raw `<a href="/#anchor">` breaks under the shared HashRouter** — it drops the surface prefix and lands on `/` (which redirects to `/employers`). Footer/anchor links must be `<Link>` with the prefix, or in-app hash routes.
- **Cross-surface links use in-app hash routes, never absolute dev ports.** The old per-app ports (web 5273, console 5173, employers 5373) no longer map to surfaces after the merge — `localhost:5173` is now the unified app root. Link with `#/console/signin`, `#/employers`, etc. Sweep: `grep -rnE "localhost:(5173|5273|5373)" src`.
- **Per-surface CSS scoping** — each surface is wrapped in `.surface-employers` / `.surface-console`; scope per-surface CSS inside that selector to prevent cross-surface bleed.
- **Design tokens are global, NOT per-surface** — they live on `:root` (light) and `:root[data-theme="dark"]` in `src/shared/styles/tokens.css`. Web surface token names are canonical. Surfaces must NOT redefine tokens on `.surface-*`.
- **Each old per-app `.gitignore` is gone** — `frontend/` has its own `.gitignore` (node_modules/dist); the root one does NOT cover it.

## Design system

- **Full standard: `docs/design-system.md`** — the self-contained catalogue (both palettes, philosophy, hard rules, verify checklist). Read it before styling anything; the bullets below are the web-side invariants only.
- **Token home:** `src/shared/styles/tokens.css` — `:root` (light defaults) + `:root[data-theme="dark"]`. All colour/spacing/font values go here; per-surface CSS uses `var(--token)` exclusively.
- **Shared primitives:** `src/shared/styles/components.css` — currently just `.ds-theme-switch` (3-way light/dark/system segmented control, consumed by `shared/theme/ThemeToggle.tsx`). The button/card/input/badge primitives once here were deleted 2026-07 (zero adopters after PR #44/#45 — each surface hand-rolled its own instead); don't reintroduce a "canonical baseline" speculatively — build one only when a real slice needs to share a component across surfaces.
- **ThemeProvider** (`src/shared/theme/`) is the **deliberate exception** to the per-surface SessionProvider rule — one global provider wraps both surfaces. It reads/writes `localStorage` key `jobify-theme`, sets `data-theme` on `<html>`, and exposes `useTheme()` → `{ theme, resolvedTheme, setTheme, toggle }`.
- **No flash on reload:** `index.html` carries an inline pre-paint `<script>` that reads `localStorage["jobify-theme"]` and sets `document.documentElement.dataset.theme` synchronously before any CSS parses — this must never be deferred or moved to a bundle.
- **console `.jc-card`** is intentionally ALWAYS-LIGHT (warm hardcoded palette for candidate preview cards). Do not tokenize it — it is a deliberate design exception, not a migration gap.
- **`frontend/styleguide/`** is a static hand-maintained swatch page — intentionally NOT wired to `tokens.css`. It is documentation, not a build artifact; update it manually when the palette changes significantly.
- **Visual language ("Match Statement" system — PR #45 employers/web, rolled out to ALL THREE surfaces):** three voices — **mono** (`--mono`) = the machine (scores, eyebrows, gauges, dashboard stats, chart/table figures), **serif italic** = the explanation/voice (the match "fit" line — the *reason*, not the caveat), **sans** = the interface. **Accent (persimmon) means ONE thing: the caveat / weakest dimension / a genuine attention state** (web marks the weakest bar via `.bar-row:has(.bar-fill.acc)`; the why-match `⌁` caveat marker is persimmon while the *reason* blockquote border is neutral; console keeps it ONLY for `.chip.acc` pending/attention states like a verification queue or pending invite) + at most one thesis emphasis per page — it is NOT decoration, NOT a data-highlight, NOT the primary-CTA colour, NOT links/active-nav/unread/focus. **Interactive = brand-blue across all three surfaces:** primary CTAs, active nav/tabs, links + hovers, unread indicators, drag-over, toggles-on, focus rings, AND console data-viz (charts, score bars, funnels, histograms, active tiles) all use `var(--brand-blue)`; filled buttons pair it with `color: var(--paper)` so type flips correctly in both themes. Destructive/error states use `var(--danger)`. **Numbers render in mono, never big serif** (scores are small mono "receipt" stamps, not billboard numerals; a number carrying a `serif` utility class is the tell — swap to `mono`). Serif headings sit at weight ~460–560 + `font-optical-sizing:auto`, never 900. When restyling, edit shared classes (`.kicker`, `.explain`, base `h1,h2,h3`, the `.btn.primary`/`.btn`/`a`/`.rail-link.active` rules) for leverage; verify BOTH themes (the only recurring dark-mode bug is a hardcoded near-light/dark colour inside an auto-inverting `var(--ink)`/`var(--paper)` panel).
- **India-first formatting lives in `src/shared/format.ts` — the single source.** `istClock` / `istDateTime` / `istDate` render **all displayed times in IST** (`Asia/Kolkata`); `inr` / `inrLakh` give `₹` with Indian 2,3 grouping / lakh shorthand. **Storage + transport stay UTC ISO** — only the render layer localizes (UTC kept in `title` tooltips on console stamps). Do not reintroduce `toISOString()`/`toUTCString()`/`getUTC*` for *display*, or `toLocaleDateString(undefined,…)` (locale-unstable) — route through this util. Surface-local money/time helpers must **delegate** to it, never re-implement (console `bits.tsx` `lakh` is a thin re-export of `inrLakh`; a hand-copied duplicate left `inrLakh` dead while the copy drifted). The `Intl.*Format` instances are **hoisted to module constants** (the ctor parses locale data — expensive on hot paths like the per-second console clock); keep them module-level, never build per-call.
- **Console account/settings:** `/console/settings` (`pages/Settings.tsx`) is the operator profile (identity + theme + session/log-out), reachable from the rail foot; **not area-gated** (any signed-in admin OR recruiter), so it sits as a plain child route, not under `RequireArea`. Operators have no résumé/DSR data, so it is deliberately lighter than a full applicant profile screen (that lived in the now-removed `web` surface; DSR/consent flows are the Flutter app's responsibility now).
